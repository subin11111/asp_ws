# ASP Final New Stack Architecture

## UAV Mission2 Path Model

The new final stack separates scan waypoints from runtime entry safety.

```text
CSV scan path:
  src/asp_final_uav/path/mission2_uav_waypoints.csv

Runtime prefix:
  generated by asp_final_uav_mission_node after Mission2 takeoff
```

The runtime transition corridor prevents the UAV from flying directly from the Mission2 origin to a
distant first scan waypoint. Each generated transition segment is limited by
`transition_corridor_max_step_m` and uses a clamped cruise altitude.

## Arrival And Anti-Stuck Logic

Mission2 waypoint arrival uses `waypoint_arrival_mode: xy_z_separate`:

```text
dxy <= waypoint_xy_tolerance_m
abs(dz) <= waypoint_z_tolerance_m
```

Transition waypoints use at least `3.0m` XY tolerance and `4.0m` Z tolerance. Yaw does not block
progression when `ignore_yaw_for_waypoint_reached` is true.

Anti-stuck behavior runs while travelling to a waypoint, not only after a hold state:

```text
WAYPOINT_STUCK_SKIP
WAYPOINT_XY_CLOSE_FORCE_ADVANCE
MARKER_TIMEOUT_CONTINUE
```

This avoids the first-waypoint loop where the UAV is close enough in XY or making no useful progress
but never satisfies a tight 3D distance threshold.

## Marker Role

Mission2 marker detection is not a dynamic route generator in the current stack. The UAV mission node
records marker IDs and publishes `MARKER_DETECTED:<id>`, but it continues following the fixed CSV path
plus runtime transition corridor. Dynamic marker-position route generation is a future feature.
